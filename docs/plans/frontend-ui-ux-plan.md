# 공사현장 문서 변환 데모 프론트엔드 Plan

기능 또는 흐름 이름:
문서 종류 선택 -> 업로드 -> 변환 로딩 -> 결과 화면 데모

이 문서는 실제 API 연동이 없는 상태에서, 공사현장 문서 변환 서비스의 핵심 흐름을 화면 데모로 빠르게 검토하기 위해 만든 실행용 plan 문서다.
목표는 사용자가 각 화면을 dev 서버에서 직접 보면서 피드백할 수 있는 클릭형 프론트엔드 데모를 만드는 것이다.

`사용자가 네 단계 화면을 dev에서 직접 보고 피드백할 수 있는 문서 변환 데모 흐름을 만든다.`

## 고정 전제

- 이번 작업은 실제 서비스 연동이 아니라 `데모` 제작이다.
- 실제 API 호출, OCR 처리, 파일 업로드 저장, 결과 파일 생성 연동은 하지 않는다.
- 모든 상태는 local mock data, seed data, sample image로 표현한다.
- 핵심 범위는 아래 4개 화면뿐이다.
  - 문서 종류 선택 화면
  - 업로드 화면
  - 변환 로딩 화면
  - 결과 화면
- 지원 문서는 `공사일보`, `자재반입 검수요청서 (철근)`, `콘크리트 반입 + CSI 입력`, `콘크리트 압축강도 + CSI 입력`, `검측 요청서`, `기타 서류(데모)`다.
- 모든 화면과 컴포넌트는 `mobile first, desktop second` 원칙으로 설계하고 구현한다.
- 구현은 화면 단위로 진행하고, 각 화면은 `구현 -> dev 확인 -> 피드백 반영` 1회까지 포함한 뒤 다음 화면으로 넘어간다.
- 이번 단계에서는 문서 관리 화면, 로그인, 권한, 실제 다운로드, 실제 변환 정확도 검증은 제외한다.

## 고정 계약 A. 문서 카탈로그

문서 선택 화면의 카드 데이터는 아래 필드를 기준으로 맞춘다.

| 필드 | 설명 |
| --- | --- |
| `type` | 내부 문서 타입 키 |
| `label` | 사용자에게 보여줄 문서명 |
| `description` | 카드에 노출할 1줄 설명 |
| `uploadGuide` | 업로드 예시 또는 안내 문구 |
| `status` | 현재 선택 가능 여부. 이번 데모에서는 모두 `available` |
| `resultLabel` | 결과 화면에서 보여 줄 문서명 |

초기 카탈로그:

| type | label | status |
| --- | --- | --- |
| `daily_report` | 공사일보 | `available` |
| `material_inspection_rebar` | 자재반입 검수요청서 (철근) | `available` |

## 고정 계약 E. 화면 구조와 전환

데모 화면 구조:

1. 문서 종류 선택 화면
2. 업로드 화면
3. 변환 로딩 화면
4. 결과 화면

기본 전환:

1. 사용자가 문서 카드를 선택한다
2. 업로드 화면에서 sample 파일 상태를 확인한다
3. CTA 클릭 시 변환 로딩 화면으로 이동한다
4. 단계형 로딩 이후 결과 화면으로 이동한다
5. 결과 화면에서 처음으로 돌아가거나 데모 CTA를 눌러 흐름을 반복한다

레이아웃 원칙:

- 모바일에서 먼저 1열 흐름으로 설계한다
- 데스크톱은 같은 정보 구조를 유지한 채 여백과 정보 밀도를 확장한다
- 모바일에서 꼭 필요한 CTA는 접지 않고 바로 노출한다

## 고정 계약 F. 디자인과 메시지 원칙

- 복잡한 장식보다 빠른 판단이 우선이다
- 레이아웃과 상호작용은 `mobile first, desktop second` 원칙을 따른다
- AI 경험이 필요한 구역은 변환 로딩 화면을 중심으로 표현한다
- 로딩 화면에서는 glow, gradation, animation을 목적 있게 사용한다
- 결과 화면에서는 완료 사실과 다음 행동이 가장 먼저 읽혀야 한다
- 오류나 실패 시나리오는 이번 데모 핵심 범위에서 제외한다

예시 메시지:

- `사진이 선명할수록 변환 결과 예시가 더 정확하게 보입니다.`
- `업로드한 자료를 바탕으로 문서 형식을 정리하고 있습니다.`
- `변환이 완료되었습니다. 결과 예시를 확인해 주세요.`
