# 공사현장 문서 자동 생성 MVP 프론트엔드 Plan

기능 또는 흐름 이름:
문서 생성 업로드 플로우 + 문서 관리 화면

이 문서는 백엔드 문서 생성 기능이 이미 준비된 상태에서, 프론트엔드가 무엇을 어떤 순서로 닫아야 하는지 빠르게 고정하기 위해 만든 실행용 plan 문서다.
설명형 기획서가 아니라, 바로 UI 구현과 검증으로 이어질 수 있는 최소 계약만 남긴다.

`사용자가 문서 종류를 선택하고 자료를 업로드한 뒤 생성 결과를 다운로드하고, 이후 문서 관리 화면에서 다시 찾을 수 있는 MVP 흐름을 닫는다.`

## 고정 전제

- 백엔드의 OCR, 문서 생성, 결과 파일 반환 기능은 이미 존재한다고 가정한다.
- 이번 MVP의 핵심 범위는 `업로드 시퀀스`와 `문서 관리 화면` 두 가지다.
- 지원 문서는 `공사일보`, `자재반입 검수요청서 (철근)`, `콘크리트 반입 + CSI 입력`, `검측 요청서`다.
- `콘크리트 압축강도 + CSI 입력`은 UI에 `준비 중`으로 노출하고 생성 흐름에는 포함하지 않는다.
- 입력은 이미지 업로드 중심으로 시작하고, 추가 파일 형식 허용 여부는 백엔드 계약 확인 후 넓힌다.
- 모든 화면과 컴포넌트는 `mobile first, desktop second` 원칙으로 설계하고 구현한다.
- 이번 단계에서는 수동 편집 기능, 삭제 기능, 관리자 기능, 상세 권한 관리는 제외한다.

## Stage 1. MVP 범위와 사용자 변화 고정

목표:
첫 버전에서 사용자가 실제로 얻게 되는 변화와 제외 범위를 먼저 고정해서, 이후 화면과 상태 설계가 흔들리지 않게 한다.

커밋 메시지:
`docs: lock MVP scope for document automation frontend`

결과물:
- `docs/plans/frontend-ui-ux-plan.md`

체크포인트:
- [ ] 첫 버전의 핵심 사용자 행동을 `문서 선택 -> 자료 업로드 -> 처리 확인 -> 결과 다운로드 -> 문서 관리 재진입`으로 고정한다
- [ ] 지원 문서와 미지원 문서를 분리해 표시 규칙을 정한다
- [ ] 백엔드가 이미 처리 로직을 제공한다는 전제를 문서에 명시한다
- [ ] MVP 제외 범위를 적어서 불필요한 구현 확장을 막는다
- [ ] 성공 기준을 `문서 생성 성공`뿐 아니라 `생성 후 재조회 가능`까지 포함한다고 적는다
- [ ] 기본 화면 설계 원칙을 `mobile first, desktop second`로 고정한다

완료 기준:
구현자가 이 문서만 보고도 이번 버전에서 꼭 만들어야 하는 흐름과 만들지 않을 기능을 구분할 수 있다.

## Stage 3. 업로드 시퀀스 화면 구조 설계

목표:
사용자가 처음 진입했을 때 어떤 순서로 문서를 만들고, 각 단계에서 무엇을 봐야 하는지 화면 구조와 상태 전환을 고정한다.

커밋 메시지:
`docs: define upload sequence architecture for document generation`

결과물:
- `docs/plans/frontend-ui-ux-plan.md`

체크포인트:
- [ ] 문서 선택 화면의 카드 정보와 CTA 동작을 고정한다
- [ ] 자료 업로드 화면의 필수 요소를 `가이드`, `업로드 영역`, `업로드 목록`, `다음 액션`으로 나눈다
- [ ] 처리 중 화면의 단계 표시 방식과 이탈 안내 문구를 고정한다
- [ ] 완료 화면의 주요 액션을 `다운로드`, `문서 관리로 이동`, `같은 문서 다시 만들기`로 고정한다
- [ ] 업로드 실패, 처리 실패, 일부 확인 필요 상태에서 보여줄 재시도 경로를 정의한다

완료 기준:
디자이너나 구현자가 문서 선택부터 완료 화면까지의 상태 변화를 별도 해석 없이 순서대로 그릴 수 있다.

## Stage 4. 문서 관리 화면 구조 설계

목표:
생성된 문서를 다시 찾고 상태를 확인하고 다운로드하는 관리 화면의 정보 구조와 상호작용을 고정한다.

커밋 메시지:
`docs: define document management table structure`

결과물:
- `docs/plans/frontend-ui-ux-plan.md`

체크포인트:
- [ ] 관리 화면 상단에 `새 문서 생성` 진입 버튼을 둔다고 명시한다
- [ ] 필터를 `문서 종류`, `상태`, `기간`, `키워드` 기준으로 고정한다
- [ ] 테이블 기본 컬럼을 정의한다
- [ ] 행 단위 액션을 `다운로드`, `상세 보기 또는 결과 보기`, `재생성`으로 정리한다
- [ ] `loading`, `empty`, `error`, `ready` 목록 상태에서 각각 어떤 화면을 보여줄지 정의한다

완료 기준:
사용자가 완료된 문서를 다시 찾고, 처리 중 문서의 상태를 확인하고, 실패 문서에 다시 대응하는 흐름이 하나의 목록 화면으로 닫혀 있다.

## Stage 5. 구현 분해와 UI 품질 기준 고정

목표:
실제 UI 구현에 들어가기 전에 컴포넌트 경계, 모바일 우선 레이아웃 기준, 상태 표현 톤을 먼저 정해서 초기 코드 구조가 흔들리지 않게 한다.

커밋 메시지:
`docs: define implementation slices and UI quality bar`

결과물:
- `docs/plans/frontend-ui-ux-plan.md`

체크포인트:
- [ ] 업로드 플로우를 `문서 카드`, `업로드 영역`, `처리 상태 패널`, `완료 요약` 단위로 나눈다
- [ ] 문서 관리 화면을 `필터 바`, `상태 배지`, `문서 테이블`, `행 액션` 단위로 나눈다
- [ ] 모든 주요 화면을 모바일 1열 레이아웃 기준으로 먼저 설계한다고 명시한다
- [ ] 데스크톱은 모바일 흐름을 유지한 채 정보 밀도와 배치를 확장하는 방향으로 정의한다
- [ ] 모바일에서 문서 관리 목록을 카드형 목록으로 우선 제공하고, 데스크톱에서 테이블로 확장할지 고정한다
- [ ] 현장 도구에 맞는 디자인 톤과 상태 배지 원칙을 고정한다
- [ ] skeleton, disabled, error, success 상태를 공통 UI 품질 기준에 포함한다

완료 기준:
구현자가 화면을 어떤 단위로 쪼개고, 모바일 기본 레이아웃 위에 데스크톱 확장을 어떻게 얹을지, 상태를 어떤 톤으로 표현해야 하는지 바로 판단할 수 있다.

## Stage 6. QA와 수용 기준 검증

목표:
실제 구현 이후 어떤 기준으로 MVP를 닫을지 마지막 검증 항목을 먼저 고정한다.

커밋 메시지:
`docs: add acceptance criteria for document generation MVP`

결과물:
- `docs/plans/frontend-ui-ux-plan.md`

체크포인트:
- [ ] 지원 문서는 선택 가능하고 미지원 문서는 선택 불가 상태로 노출되는지 확인한다
- [ ] 파일이 0개일 때 다음 단계로 진행되지 않는지 확인한다
- [ ] 업로드 후 처리 중 상태가 사용자에게 보이고 이탈 안내가 있는지 확인한다
- [ ] 생성 완료 후 즉시 다운로드와 문서 관리 이동이 모두 가능한지 확인한다
- [ ] 문서 관리 화면에서 상태별 UI와 기본 액션이 정상 노출되는지 확인한다
- [ ] 모바일 기본 폭에서 업로드와 결과 조회가 깨지지 않는지 확인한다
- [ ] 데스크톱 폭에서도 모바일 기준 흐름이 자연스럽게 확장되는지 확인한다

완료 기준:
행복 경로, 실패 경로, 미지원 문서 차단, 모바일 우선 설계와 데스크톱 확장 검증까지 포함한 MVP 수용 기준이 명확하게 체크 가능하다.

## 고정 계약 A. 문서 카탈로그

문서 선택 화면의 카드 데이터는 아래 필드를 기준으로 맞춘다.

| 필드 | 설명 |
| --- | --- |
| `type` | 내부 문서 타입 키 |
| `label` | 사용자에게 보여줄 문서명 |
| `description` | 카드에 노출할 1줄 설명 |
| `uploadGuide` | 업로드 예시 또는 안내 문구 |
| `status` | `available` 또는 `coming_soon` |
| `outputFormat` | 결과물 형식 예시 (`xlsx`) |

초기 카탈로그:

| type | label | status | outputFormat |
| --- | --- | --- | --- |
| `daily_report` | 공사일보 | `available` | `xlsx` |
| `material_inspection_rebar` | 자재반입 검수요청서 (철근) | `available` | `xlsx` |
| `concrete_delivery_csi` | 콘크리트 반입 + CSI 입력 | `available` | `xlsx` |
| `concrete_strength_csi` | 콘크리트 압축강도 + CSI 입력 | `coming_soon` | `xlsx` |
| `inspection_request` | 검측 요청서 | `available` | `xlsx` |

## 고정 계약 B. 업로드, 처리, 결과 데이터

업로드 요청 최소 필드:

| 필드 | 설명 |
| --- | --- |
| `documentType` | 선택한 문서 타입 |
| `files` | 업로드 파일 배열 |
| `metadata` | 선택적 부가 정보. MVP에서는 비워 둘 수 있음 |

처리 작업 상태:

| 상태 | 의미 |
| --- | --- |
| `idle` | 아직 시작 전 |
| `files_attached` | 파일 첨부 완료 |
| `uploading` | 파일 전송 중 |
| `processing` | 서버 문서 생성 중 |
| `completed` | 생성 완료 |
| `partial_success` | 결과는 나왔지만 확인 필요 |
| `failed` | 생성 실패 |

문서 관리 목록 상태:

| 상태 | 의미 |
| --- | --- |
| `loading` | 목록 불러오는 중 |
| `empty` | 생성된 문서 없음 |
| `ready` | 목록 표시 가능 |
| `error` | 목록 조회 실패 |

결과 응답 최소 필드:

| 필드 | 설명 |
| --- | --- |
| `jobId` | 생성 작업 식별자 |
| `documentType` | 문서 타입 |
| `status` | 처리 결과 상태 |
| `downloadUrl` | 결과 파일 다운로드 URL |
| `fileName` | 다운로드 파일명 |
| `warningMessage` | 일부 확인 필요 시 노출 문구 |
| `createdAt` | 생성 시각 |

업로드 validation 규칙:

- `coming_soon` 문서는 선택 또는 제출할 수 없다.
- 업로드 파일이 1개도 없으면 제출할 수 없다.
- 허용되지 않은 파일 형식은 업로드 직후 차단한다.
- 업로드 중에는 중복 제출을 막는다.
- `partial_success` 상태에서는 경고 문구와 함께 다운로드를 허용한다.

## 고정 계약 C. 화면 구조와 상태 흐름

업로드 시퀀스 화면 구조:

1. 문서 선택 화면
2. 자료 업로드 화면
3. 처리 중 화면
4. 완료 화면

업로드 시퀀스 기본 전환:

1. 사용자가 문서 카드를 선택한다
2. 업로드 화면에서 자료를 첨부하고 제출한다
3. 처리 중 화면에서 단계형 진행 상태를 보여 준다
4. 완료 시 결과 다운로드와 문서 관리 이동을 함께 제공한다
5. 실패 시 같은 화면 안에서 재시도 또는 이전 단계 복귀를 제공한다

문서 관리 화면 구조:

- 상단 제목
- 새 문서 생성 버튼
- 필터 바
- 상태 포함 테이블 또는 모바일 카드 목록
- 행 액션

레이아웃 원칙:

- 모바일에서 먼저 1열 흐름으로 설계한다.
- 데스크톱은 같은 정보 구조를 유지한 채 2열 이상 배치, 여유 있는 여백, 테이블 밀도 확장으로 대응한다.
- 모바일에서 꼭 필요한 액션은 접지 않고 바로 노출한다.

문서 관리 기본 컬럼:

- 문서 종류
- 생성 요청 일시
- 처리 상태
- 업로드 파일 수
- 결과물 형식
- 다운로드
- 상세 보기 또는 결과 보기

## 고정 계약 D. 디자인과 메시지 원칙

- 복잡한 장식보다 빠른 판단이 우선이다.
- 레이아웃과 상호작용은 `mobile first, desktop second` 원칙을 따른다.
- 상태 배지는 강한 대비로 구분한다.
- 처리 중에는 퍼센트보다 단계 문구 중심으로 불안을 줄인다.
- 완료 화면에서는 `다운로드`를 가장 먼저 보이게 둔다.
- 오류 메시지는 원인과 다음 행동을 함께 말한다.
- 모바일에서는 업로드와 결과 확인이 손가락 조작 기준으로 쉬워야 한다.

예시 메시지:

- `사진이 흐리면 OCR 정확도가 낮아질 수 있습니다.`
- `문서 생성이 진행 중입니다. 이 화면을 떠나도 문서 관리에서 다시 확인할 수 있습니다.`
- `일부 항목을 정확히 읽지 못했습니다. 결과를 확인해 주세요.`
