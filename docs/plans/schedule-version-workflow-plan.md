# 공정표 버전 관리 UX Plan

기능 또는 흐름 이름:
기준 공정표 -> 작업본 생성 -> 여러 작업본 비교 -> 기준 공정표로 반영

이 문서는 `main/sub/fork`처럼 개발자에게는 익숙하지만 현장 사용자에게는 낯선 공정표 버전 개념을, 화면에서 자연스럽게 이해할 수 있는 UX로 바꾸기 위한 실행용 plan 문서다.
목표는 기준 공정표를 안전하게 읽기 전용으로 유지하면서, 여러 작업본을 만들어 검토하고 선택한 작업본을 기준 공정표로 반영하는 흐름을 고정하는 것이다.

`사용자는 기준 공정표를 안전하게 보존한 채 여러 작업본을 만들고 비교한 뒤, 선택한 작업본을 기준 공정표로 반영할 수 있다.`

## 고정 전제

- backend/API 내부 용어가 `main`, `sub`, `fork`이더라도 UI에서는 그대로 노출하지 않는다.
- `main`은 `기준 공정표`로 표현한다.
- `sub`는 `작업본` 또는 문맥상 `수정안`으로 표현한다.
- `fork`는 `작업본 만들기`로 표현한다.
- `main 교체/promote`는 `기준 공정표로 반영`으로 표현한다.
- 기준 공정표는 읽기 전용이며, 작업 생성/수정/삭제/연결/마일스톤 편집을 할 수 없다.
- 작업본은 여러 개 만들 수 있으며, 각 작업본은 독립적으로 수정할 수 있다.
- 여러 작업본은 비교와 검토를 위해 보관될 수 있다.
- 기준 공정표로 반영하기 전에는 변경사항 검토 단계를 거친다.
- 삭제, 기준 반영, 읽기 전용 전환처럼 되돌리기 어려운 행동은 확인 단계를 둔다.

## Stage 1. 용어와 mental model 고정 [completed]

목표:
사용자가 개발자식 버전 개념을 몰라도 화면을 이해할 수 있도록, UI에서 사용할 이름과 설명 문장을 먼저 고정한다.

커밋 메시지:
`docs: define schedule version ux language`

결과물:
- `docs/plans/schedule-version-workflow-plan.md`

체크포인트:
- [x] `main`을 화면에서 `기준 공정표`로만 표현한다
- [x] `sub`를 화면에서 `작업본`으로만 표현한다
- [x] `fork` 액션을 `작업본 만들기`로 표현한다
- [x] 기준 교체 액션을 `기준 공정표로 반영`으로 표현한다
- [x] 기준 공정표 설명을 `현장에서 공식으로 보는 읽기 전용 공정표`로 고정한다
- [x] 작업본 설명을 `기준 공정표에서 복사해 만든 수정 가능한 안`으로 고정한다

완료 기준:
기획자, 디자이너, 구현자가 `main/sub/fork`를 직접 쓰지 않고도 같은 화면 카피와 같은 개념으로 대화할 수 있다.

## Stage 2. 버전 데이터와 상태 계약 정의 [completed]

목표:
기준 공정표와 여러 작업본을 화면에서 구분하고, 어떤 버전이 수정 가능한지 판단할 수 있는 최소 데이터 계약을 정한다.

커밋 메시지:
`docs: define schedule version state contract`

결과물:
- 공정표 버전 UI model 정의
- 필요한 backend API 변경 질문 목록
- 기존 공정표 API 타입 업데이트 후보
- frontend mock history model 정의

체크포인트:
- [x] 버전 목록 항목에 `id`, `name`, `isMain`, `createdAt`, `updatedAt`이 필요하다고 정의한다
- [x] UI model에서 `isMain=true`이면 `기준 공정표`, 아니면 `작업본`으로 표시한다고 정의한다
- [x] 선택 중인 버전이 기준 공정표이면 `readOnly=true`로 계산한다고 정의한다
- [x] 작업본은 여러 개 존재할 수 있다고 정의한다
- [x] 작업본 이름 예시는 `260402_작업본1`, `260402_작업본2`, `260403_우천반영안`처럼 날짜 prefix를 넣는다고 고정한다
- [x] 기본 작업본 이름은 `YYMMDD_작업본N` 형식으로 생성한다고 정의한다
- [x] 기준 공정표 반영 API가 기존 main을 교체하는지, 선택 버전의 `isMain`을 바꾸는지 backend와 확인한다
- [x] 기준 공정표를 반영한 뒤 기존 기준 공정표는 삭제하지 않고 `기준 공정표 history`로 남긴다고 정의한다
- [x] backend history API는 후속으로 추가하고, frontend에서는 먼저 mock history가 있는 것처럼 보여준다고 정의한다
- [x] history 항목에는 `id`, `name`, `promotedAt`, `promotedBy`, `sourceDraftName`, `summary`가 필요하다고 정의한다
- [x] history 목록 entry point는 기준 공정표 chip 또는 버전 dropdown 안에 `기준 공정표 이력 보기`로 둔다고 정의한다
- [x] 기준 공정표 read-only 시각 표현은 bar 테두리 제거, 일반 weight 텍스트, 낮은 채도, 잠금 chip을 조합한다고 정의한다

완료 기준:
화면이 버전 목록을 받아서 `기준 공정표`, `작업본`, `읽기 전용`, `수정 가능` 상태를 별도 해석 없이 계산할 수 있다.

목표:
사용자가 현재 보고 있는 공정표가 기준 공정표인지 작업본인지 즉시 알 수 있고, 기준 공정표에서 왜 편집이 막히는지 자연스럽게 이해하게 만든다.

커밋 메시지:
`feat: add schedule version selector read only state`

결과물:

- 기준 공정표 읽기 전용 chip
- 작업본 수정 가능 chip
- 편집 불가 상태 UI 규칙

체크포인트:
- [x] 공정표 상단에 현재 버전 chip을 추가한다
- [x] 기준 공정표 선택 시 `기준 공정표 · 읽기 전용` chip을 보여준다
- [x] 작업본 선택 시 `작업본 · 수정 가능` chip을 보여준다
- [x] 기준 공정표에서는 bar drag, resize, rename, delete를 막는다
- [x] 기준 공정표에서는 작업 연결 생성/삭제를 막는다
- [x] 기준 공정표에서는 마일스톤 생성/수정/삭제를 막는다
- [x] 기준 공정표 bar는 기존처럼 테두리를 유지한다
- [x] 기준 공정표 bar 텍스트는 기존처럼 bold weight를 유지한다
- [x] 기준 공정표 작업 연결은 hover 전까지 낮은 대비로 보여준다
- [x] 좌측 header에는 현재 공정표 이름을 노출한다
- [x] 기준 공정표에서 편집 시도 시 `기준 공정표는 직접 수정할 수 없어요. 작업본을 만들어 수정해 주세요.` toast를 보여준다
- [x] 기준 공정표 toolbar에는 `작업본 만들기` CTA를 우선 노출한다

완료 기준:
사용자가 기준 공정표 화면에서 실수로 공식 일정을 수정할 수 없고, 수정하려면 작업본을 만들어야 한다는 흐름을 바로 이해한다.

## Stage 4. 작업본 생성과 관리 흐름 구현

목표:
사용자가 기준 공정표나 기존 작업본에서 새 작업본을 만들고, 여러 작업본을 이름으로 구분해 관리할 수 있게 한다.

커밋 메시지:
`feat: add schedule draft version workflow`

결과물:
- `작업본 만들기` action
- 작업본 이름 입력 UI
- 작업본 목록 UI
- 작업본 이름 변경/삭제 메뉴

체크포인트:
- [x] `작업본 만들기` 클릭 시 현재 선택 버전을 복사한다고 정의한다
- [x] 기본 작업본 이름은 `YYMMDD_작업본N` 형식으로 제안한다
- [x] 사용자가 작업본 이름을 직접 수정할 수 있게 한다
- [x] 작업본 생성 후 새 작업본으로 자동 이동한다
- [x] 작업본 목록에서 기준 공정표를 항상 최상단에 고정한다
- [x] 작업본은 생성 순서 또는 최근 수정 순서로 정렬한다
- [x] 작업본 우클릭 또는 더보기 메뉴에서 이름 변경을 제공한다
- [x] 작업본 삭제 전 확인 dialog를 보여준다
- [x] 기준 공정표는 삭제 메뉴를 보여주지 않는다

완료 기준:
사용자가 여러 작업본을 만들고, 이름을 붙이고, 필요 없는 작업본을 정리할 수 있다.

## Stage 5. 작업본 변경사항 검토와 비교 UX 설계 [completed]

목표:
여러 작업본을 비교하거나 기준 공정표로 반영하기 전에, 무엇이 달라졌는지 사용자가 빠르게 판단할 수 있게 한다.

커밋 메시지:
`feat: add schedule draft review summary`

결과물:
- 기준 공정표와 작업본 비교 toggle
- 차트 위 기준 공정표 ghost overlay
- 변경 유형 chip 없이 기준 위치를 아주 옅게 겹쳐 보는 chart review mode

체크포인트:
- [x] 작업본 상단에 비교 toggle을 둔다
- [x] 변경사항 요약을 `작업 추가`, `작업 변경`, `작업 삭제`, `작업 연결 변경`, `마일스톤 변경`으로 나눈다
- [x] 변경사항은 리스트 대신 차트 ghost overlay로 보여준다
- [x] bar 변경은 날짜/기간/이름/row 이동 중 어떤 변경인지 구분한다
- [x] 작업 연결 변경은 source/target/gap 변경을 구분한다
- [x] 마일스톤 변경은 생성/이동/이름 변경/삭제를 구분한다
- [x] 비교 UI에서 기준 공정표 위치를 아주 흐린 ghost overlay로 구분한다
- [x] 여러 작업본을 나란히 비교하는 기능은 1차에서는 작업본별 기준 비교 toggle로 시작하고, side-by-side canvas는 후속 단계로 둔다

완료 기준:

## Stage 6. 기준 공정표로 반영 흐름 구현

목표:
검토가 끝난 작업본을 기준 공정표로 안전하게 반영하고, 사용자가 반영 결과를 명확히 확인할 수 있게 한다.

커밋 메시지:
`feat: promote schedule draft to baseline`

결과물:
- `기준 공정표로 반영` action
- 반영 전 확인 dialog
- 반영 완료 toast 또는 result state
- 반영 후 버전 목록 갱신
- 기준 공정표 history mock panel

체크포인트:
- [ ] 작업본에서만 `기준 공정표로 반영` CTA를 보여준다
- [ ] 기준 공정표에서는 `기준 공정표로 반영` CTA를 숨긴다
- [ ] 반영 전 변경사항 요약을 다시 보여준다
- [ ] 확인 dialog 문구를 `이 작업본을 기준 공정표로 반영할까요?`로 고정한다
- [ ] dialog 안에 `기존 기준 공정표는 더 이상 기준으로 쓰이지 않아요.` 같은 영향 설명을 넣는다
- [ ] 반영 성공 시 선택 버전을 새 기준 공정표로 갱신한다
- [ ] 반영 성공 시 이전 기준 공정표를 history 목록에 추가한 것처럼 보여준다
- [ ] history panel에서 과거 기준 공정표 이름, 반영일, 반영한 작업본 이름, 변경 요약을 보여준다
- [ ] 반영 성공 시 `기준 공정표로 반영했어요.` toast를 보여준다
- [ ] 반영 실패 시 기존 선택과 작업본 상태를 유지하고 error toast를 보여준다

완료 기준:
사용자가 실수 없이 작업본을 기준 공정표로 반영할 수 있고, 반영 후 어떤 버전이 기준인지 즉시 확인할 수 있다.

## Stage 7. QA와 수용 기준 검증

목표:
기준 공정표/작업본 개념이 실제 화면에서 헷갈리지 않고, 편집 제한과 반영 흐름이 안전하게 동작하는지 검증한다.

커밋 메시지:
`test: verify schedule version workflow`

결과물:
- 수용 기준 체크리스트
- 수동 QA 시나리오
- 필요한 경우 컴포넌트 또는 state 단위 테스트

체크포인트:
- [ ] 기준 공정표에서 모든 편집 액션이 막히는지 확인한다
- [ ] 기준 공정표에서 편집 시도 시 작업본 생성 안내가 보이는지 확인한다
- [ ] 기준 공정표에서 작업본을 만들면 새 작업본으로 이동하는지 확인한다
- [ ] 작업본을 여러 개 만들고 각각 다른 이름으로 구분되는지 확인한다
- [ ] 작업본 수정 후 변경사항 요약 count가 맞는지 확인한다
- [ ] 작업본 삭제 전 확인 dialog가 뜨는지 확인한다
- [ ] 기준 공정표 삭제 메뉴가 노출되지 않는지 확인한다
- [ ] 작업본을 기준 공정표로 반영하면 새 기준 상태와 읽기 전용 상태가 적용되는지 확인한다
- [ ] 반영 실패 시 작업본이 사라지거나 선택이 깨지지 않는지 확인한다

완료 기준:
사용자가 `기준 공정표는 안전하게 보존되고, 작업본에서만 수정하며, 검토 후 기준으로 반영한다`는 흐름을 별도 설명 없이 화면에서 이해하고 완료할 수 있다.
