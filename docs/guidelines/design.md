# 디자인 가이드: 뉴트럴 베이스 + AI 블루 인터페이스

## 1. 개요와 디자인 방향

이 문서는 서비스 전반에 적용할 수 있는 기본 디자인 원칙을 정리한 가이드다.
목표는 장식적인 화면보다 빠른 판단과 명확한 행동 유도를 우선하는 인터페이스를 만드는 것이다.

전체 톤은 다음 방향을 따른다.

- 회색 기반의 뉴트럴 캔버스
- 프라이머리 컬러 `#1e1888` 중심의 또렷한 액션 계층
- 강한 대비와 분명한 정보 구조
- 기본적으로는 평면적이고 실무형인 운영 화면
- 대시보드와 공정표처럼 정보 밀도가 높은 데스크톱 운영 화면
- 문서 생성처럼 모바일에서 먼저 완성되고 데스크톱으로 확장되는 작업 흐름
- AI 전용 영역에서는 glow, gradation, animation을 활용한 경험 강화

화면은 지나치게 화려하거나 가벼운 인상보다, 또렷하고 신뢰감 있으며 실무 도구처럼 느껴져야 한다.
다만 AI 기능을 소개하거나 처리 중 상태를 전달하는 구역은 조금 더 살아 있는 시각 경험을 가져도 된다.

`이 화면에서 지금 무엇을 해야 하는지 즉시 이해할 수 있어야 하며, AI 기능이 작동 중일 때는 그 경험이 분명하게 느껴져야 한다.`

## 2. 핵심 원칙

### 2.1 뉴트럴 베이스 + 단일 프라이머리 블루

기본 색상 체계는 검정, 회색, 흰색을 중심으로 유지하되, 프라이머리 컬러로 `#1e1888`을 사용한다.

- 프라이머리 컬러는 주요 CTA, 선택 상태, 포커스, 핵심 인터랙션 강조에 사용한다
- 파랑 외의 다중 강조색은 기본 체계로 확장하지 않는다
- 강조는 가능한 한 색상 하나보다 대비, 두께, 크기, 간격과 함께 해결한다

강조 방법 예시:

- 더 진한 배경 채움
- 더 무거운 글자 두께
- 더 큰 크기 차이
- 더 굵은 외곽선
- 핵심 액션에 프라이머리 블루 적용
- 핵심 액션 주변 여백 축소

### 2.2 기본은 절제, AI 영역은 예외 허용

서비스 전체 UI는 기본적으로 절제된 평면형 언어를 사용한다.
하지만 AI 기능을 직접 경험하는 구역에서는 glow, gradation, animation을 예외적으로 사용할 수 있다.

기본 화면에서 지양하는 표현:

- 장식 목적의 강한 그라디언트
- 과한 글로우
- 유리 질감
- 블러 기반 유리 효과
- 메탈릭 광택

AI 영역에서 허용되는 표현:

- 프라이머리 블루 기반의 그라디언트
- 처리 상태를 보여 주는 은은한 glow
- 생성, 분석, 변환, 추천 상태를 전달하는 애니메이션

AI 시각 표현은 아래 조건을 만족해야 한다.

- AI 기능과 직접 관련된 구역에서만 사용한다
- 읽기와 조작을 방해하지 않는다
- 화면 전체를 덮지 않고 국소적으로 사용한다
- 의미 없는 장식 반복이 아니라 상태 전달 역할을 가진다

### 2.3 기본 배경은 회색 캔버스

앱의 기본 캔버스는 밝은 회색 바탕을 사용한다.
배경은 패턴 없이 단색으로 유지하며, 정보 읽기 영역을 방해해서는 안 된다.

원칙:

- 정보가 많은 카드와 패널은 흰색 단색 면 위에 둔다
- 기본 앱 배경은 단색 회색 캔버스로 고정한다
- 배경 효과가 필요할 때도 패턴 대신 면 분리, 대비, 여백으로 해결한다
- AI 영역의 시각 효과도 콘텐츠 가독성을 해치지 않아야 한다

### 2.4 강한 경계 규칙

레이아웃은 주로 여백과 면 분리로 구성하되, 중요한 상호작용 요소는 분명한 경계가 있어야 한다.

- 카드와 패널은 흰색 면으로 분리한다
- 주요 액션은 프라이머리 블루 채움으로 구분한다
- 일반 표면은 얇은 회색 외곽선을 사용한다
- 회색 캔버스와 흰색 콘텐츠 면 사이에 충분한 톤 차이를 둔다

너무 약한 구분선이나 흐릿한 전환은 지양한다.

### 2.5 AI 경험 표현 원칙

AI 영역의 시각 언어는 "화려함"보다 "작동 중인 지능형 시스템"을 느끼게 하는 방향이어야 한다.

권장되는 AI 표현:

- 느린 펄스
- 짧은 shimmer
- 진행 방향이 느껴지는 gradient sweep
- 집중된 blue glow
- 결과 도착 시 짧은 reveal animation

피해야 하는 AI 표현:

- 계속 반짝이는 무의미한 효과
- 본문 뒤에서 강하게 움직이는 배경
- 버튼, 카드, 텍스트가 동시에 과하게 움직이는 상태
- 장시간 반복되어 피로감을 주는 모션

## 3. 색상 시스템과 토큰

이 팔레트는 중립적이고 신뢰감 있는 베이스 위에, 프라이머리 블루가 명확한 액션 신호를 주는 구조를 따른다.

### 3.1 기본 토큰

```css
--canvas: #f4f4f4;
--surface-1: #ffffff;
--surface-2: #ffffff;
--surface-3: #efefef;
--surface-inverse: #111111;
--primary: #1e1888;
--primary-hover: #19146f;
--primary-pressed: #141058;
--primary-soft: #eeedff;
--primary-outline: rgba(30, 24, 136, 0.24);
--primary-glow: rgba(30, 24, 136, 0.30);
--primary-gradient-start: #1e1888;
--primary-gradient-end: #4b44d4;
--ink-strong: #000000;
--ink: #171717;
--ink-muted: #5c5c5c;
--ink-faint: #8b8b8b;
--outline-strong: rgba(0, 0, 0, 0.72);
--outline-soft: rgba(0, 0, 0, 0.14);
--shadow-soft: 0 10px 24px rgba(0, 0, 0, 0.08);
```

### 3.2 표면 계층

- 캔버스: `--canvas`
- 기본 섹션 배경: `--surface-1`
- 주요 카드: `--surface-2`
- 보조 패널 또는 인셋: `--surface-3`
- 주요 액션 면: `--primary`
- 강한 다크 표면: `--surface-inverse`
- AI 강조 면: `--primary-soft` 또는 `--primary-gradient-start -> --primary-gradient-end`

### 3.3 대비 규칙

- Primary CTA: `#1e1888` 배경, 흰색 텍스트
- Secondary CTA: 흰색 배경, 검정 텍스트 또는 프라이머리 외곽선
- 일반 칩: 흰색 또는 연회색 배경, 어두운 텍스트
- 선택된 칩: 프라이머리 블루 배경, 흰색 텍스트
- 현재 활성 항목: 프라이머리 외곽선 + 강한 제목 두께
- 비활성 항목: 회색 배경 + 낮은 대비 텍스트
- AI 상태 배지: 연한 블루 면 + 진한 텍스트 또는 프라이머리 채움

### 3.4 상태 표현 규칙

성공, 실패, 진행 중 같은 상태를 색상 하나에만 의존하지 않는다.
프라이머리 블루는 `주요 행동`, `선택`, `AI 존재감`을 강화하는 용도로 사용하고, 상태 의미 전체를 대신하지 않는다.

상태를 표현할 때는 아래 수단을 함께 사용한다.

- 텍스트 문구
- 아이콘
- 외곽선 스타일
- 채움 반전
- 뱃지, 라벨, 좌측 바 같은 구조적 표시

의미가 색상만으로 전달되면 안 된다.

## 4. 타이포그래피

타이포그래피는 이 시스템에서 가장 중요한 계층 도구 중 하나다.

### 4.1 폰트 조합

- 디스플레이와 헤드라인: `Space Grotesk`
- 본문과 라벨: `IBM Plex Sans`

구현 환경에서 해당 폰트를 바로 쓰기 어렵다면, 같은 역할 구조를 유지하는 대체 폰트를 사용한다.

### 4.2 두께 계층

- 대형 타이틀 또는 핵심 수치: `700-800`
- 카드 제목: `700-800`
- CTA 라벨: `800`
- 본문: `400-500`
- 보조 라벨: `600-700`

### 4.3 크기 원칙

- 대표 타이틀은 우아함보다 단호함이 느껴져야 한다
- 수치, 시간, 상태는 작더라도 무게감 있게 보여야 한다
- 보조 문구는 이탤릭이나 장식보다 크기와 대비로 차이를 만든다
- 섹션 제목은 별도 구분선 없이도 경계를 만들어야 한다
- AI 영역 헤드라인은 필요할 때만 약한 컬러 강조나 reveal 모션을 더할 수 있다

### 4.4 역할 기반 타입 체계

텍스트 크기는 화면마다 임의로 정하지 말고, 역할 기준으로 먼저 배정한다.

- `display-xl`: 완료 화면 또는 가장 강한 메시지
- `display-lg`: 주요 페이지 타이틀
- `display-md`: 작업 화면의 대표 제목
- `title-lg`: 카드 또는 섹션 헤드라인
- `title-md`: 단계 제목, 리드 문구
- `title-sm`: 보조 패널 제목
- `body-lg`: 카드 안 주요 본문
- `body-md`: 설명용 본문
- `body-sm`: 헬퍼 문구, 메타 정보
- `label-md`: 기본 버튼과 컨트롤 라벨
- `label-sm`: 작은 유틸리티 컨트롤
- `label-xs`: 오버라인, 보조 라벨

같은 역할의 텍스트는 다른 화면에서도 같은 타입 계층을 유지한다.

## 5. 배경 캔버스 규칙

배경은 단색 회색 캔버스를 기본으로 사용한다.
리듬과 계층은 패턴이 아니라 면 분리, 대비, 여백, 카드 구조로 만든다.

### 5.1 기본 배경 규칙

- 전역 앱 배경은 단색 회색으로 유지한다
- 읽기 영역과 입력 영역은 흰색 단색 면 위에 둔다
- 시각적 복잡도는 배경이 아니라 콘텐츠 구조에서 만든다
- AI 영역의 시각 효과도 단색 배경 위에서 더 또렷하게 읽혀야 한다

### 5.2 배경 사용 원칙

- 배경에 점, 격자, 질감 패턴을 넣지 않는다
- 강한 시각 효과가 필요할 때는 국소적인 gradient backplate나 glow를 사용한다
- 일반 업무 화면의 배경은 항상 차분하고 안정적으로 유지한다
- 화면 전체 배경이 콘텐츠보다 먼저 눈에 들어오지 않게 한다

## 6. 레이아웃, 표면, 깊이

### 6.1 표면 구조

깊이는 분위기 효과가 아니라, 평면적 적층으로 만든다.

- canvas -> section -> card -> inset
- 각 레벨은 한눈에 구분되어야 한다
- 그림자는 보조 수단일 뿐, 주된 구분 수단이 되어서는 안 된다

### 6.2 화면 유형별 반응형 원칙

현재 앱은 두 가지 화면 유형을 함께 가진다.

- 운영형 데스크톱 화면: 대시보드, 공정표
- 모바일 우선 작업 플로우: 문서 선택, 업로드, 로딩, 결과, 생성 문서 목록

운영형 데스크톱 화면은 `1180px` 이상의 업무 도구로 보고, 무리하게 모바일 화면으로 접지 않는다.
대시보드와 공정표는 넓은 캔버스에서 정보를 비교하고 조작하는 것이 핵심이므로 `min-width`를 허용한다.

문서 생성 플로우는 모바일을 기본 경험으로 유지한다.
모바일에서는 `42rem` 내외의 단일 컬럼 셸을 쓰고, 태블릿 이상에서는 `52rem`까지 확장한다.
데스크톱에서는 기존 모바일 디자인 언어를 보존하되, `DesktopAppHeader`, `76rem` 셸, 2열 레이아웃을 사용할 수 있다.

반응형 확장 원칙:

- 모바일에서는 1열 흐름을 기본으로 한다
- 핵심 액션은 접히지 않고 바로 보이게 둔다
- 데스크톱은 같은 정보 구조를 유지한 채 여백, 열 수, 밀도를 확장한다
- 문서 생성 데스크톱 화면은 왼쪽 작업 영역과 오른쪽 보조/결과 영역을 하나의 row 덩어리로 읽히게 한다
- 같은 row 안의 outer box는 가능하면 같은 높이로 stretch 한다
- 오른쪽 보조 리스트는 내용이 많으면 내부 스크롤을 사용하고, outer box 자체를 늘려 레이아웃을 깨지 않는다
- 모바일과 데스크톱이 다른 제품처럼 느껴지지 않게 한다

### 6.3 코너 반경 규칙

- 카드: `16px-24px`
- 입력창과 버튼: `14px-18px`
- 칩: pill 형태 허용

둥근 모서리는 귀엽기보다 단단하게 느껴져야 한다.

일회성 값 대신 등급별로 반경을 쓴다.

- `card radius`: 주요 카드와 큰 셸
- `panel radius`: 카드 안 보조 패널
- `control radius`: 버튼, 토글, 리스트 아이템
- `inset radius`: 작은 아이콘 버튼, 조밀한 헬퍼 표면
- `pill radius`: 칩, 뱃지, 캡슐형 상태 표시

주요 CTA 버튼은 특별한 이유가 없으면 기본적으로 pill 형태로 만들지 않는다.

### 6.4 그림자와 glow 규칙

일반 화면의 그림자는 절제해서 사용한다.

- 먼저 외곽선과 톤 차이로 구분한다
- 그림자가 필요할 때도 짧고 중립적으로 쓴다
- 일반 업무 화면에서는 색이 섞인 그림자를 남용하지 않는다

AI 영역에서는 아래 예외를 허용한다.

- 프라이머리 블루 기반 glow
- 중립 그림자 위에 얹는 얇은 외곽 glow
- 처리 중 상태를 알려 주는 약한 발광 효과

다만 glow는 콘텐츠보다 앞에 나오면 안 되며, 조작 요소를 흐리게 만들어서도 안 된다.

### 6.5 데스크톱 운영 화면 레이아웃

대시보드와 공정표는 현장 운영자가 한 화면에서 비교, 판단, 조작하는 화면이다.
카드형 정보 구조를 쓰되, 정보 밀도를 과도하게 낮추지 않는다.

데스크톱 운영 화면 규칙:

- 페이지 배경은 `#f5f5f6` 또는 `--canvas` 계열의 단색 회색을 사용한다
- 상단에는 `DesktopAppHeader`를 사용하고, 본문은 `76rem` 셸 또는 작업 성격에 맞는 full-width 작업 셸을 사용한다
- 대시보드 카드는 흰색 면, 얇은 회색 외곽선, `0.8rem-1.3rem` 반경을 기본으로 한다
- 한 row 안에서 비교되는 카드들은 높이를 맞춰 하나의 덩어리로 읽히게 한다
- 카드 안 리스트가 길어질 때는 카드 높이를 무한히 늘리지 말고 내부 스크롤을 사용한다
- 카드 안 child item은 위에서부터 쌓고, 남는 세로 공간 때문에 카드가 강제로 늘어나 보이면 안 된다
- count나 meta 값은 꼭 상호작용 요소가 아니라면 pill box로 만들지 않고 일반 텍스트로 둔다
- 제목, 리스트, CTA는 각각 명확한 시작선을 가져야 하며, 좌우 카드의 헤더 패딩은 가능하면 맞춘다

### 6.6 문서 생성 플로우 레이아웃

문서 생성 플로우는 모바일에서 가장 먼저 완성되어야 한다.
데스크톱에서는 모바일 카드, 버튼, 헤더 리듬을 보존한 채 화면 여백과 보조 정보를 확장한다.

문서 생성 화면 규칙:

- 모바일 헤더는 로고 또는 back button 중심의 단순한 상단 바를 사용한다
- 모바일 페이지는 흰색 배경을 기본으로 하며, bottom CTA가 분명히 보이게 한다
- 문서 선택 카드는 한 줄씩 세로로 나열한다
- 데스크톱 문서 선택 화면은 왼쪽 생성 대상 리스트와 오른쪽 생성 문서 리스트를 같은 row의 outer box로 맞춘다
- 오른쪽 생성 문서 리스트는 문서가 없으면 `생성된 문서가 없어요` 같은 muted empty state를 보여준다
- 문서가 몇 개 없으면 위에서부터 정렬하고, `전체 보기` 버튼과 충분한 간격을 유지한다
- 문서가 많으면 리스트 영역에만 스크롤을 붙인다
- 업로드/결과/생성 문서 목록 화면은 모바일 셸을 유지하되 태블릿 이상에서 폭만 확장한다

## 7. 컴포넌트 가이드

### 7.1 앱 셸

- 평평한 단색 회색 캔버스
- 콘텐츠는 흰색 둥근 박스로 읽혀야 한다
- 정보 밀도가 높은 화면일수록 충분한 여백을 둔다
- AI 섹션이 들어갈 때만 국소적 블루 gradient backplate를 둘 수 있다
- 데스크톱 운영 화면은 `DesktopAppHeader`와 본문 셸을 분리한다
- 문서 생성 모바일 화면은 자체 모바일 헤더를 유지하고, 데스크톱 확장 시에만 `DesktopAppHeader`를 추가한다

### 7.2 상단 바와 내비게이션

- 조밀하더라도 통제된 인상을 유지한다
- 정체성은 색조보다 텍스트와 배치에서 만든다
- 컨트롤은 도구처럼 보여야 하며 과하게 장식하지 않는다
- 상단 주요 내비게이션의 활성 상태는 진한 텍스트와 하단 underline으로 충분하다
- 탭, 토글, 작업 컨트롤의 활성 상태는 프라이머리 블루나 프라이머리 외곽선으로 더 분명히 표현할 수 있다

### 7.3 Primary CTA

- `#1e1888` 배경
- 흰색 텍스트
- 기본 상태는 단색 채움 우선
- 눌림 상태는 크기 변화나 톤 변화로 표현

Primary, Secondary, Utility 버튼은 하나의 공통 제어 체계를 공유한다.

- 같은 높이 체계
- 같은 좌우 패딩 체계
- 같은 라벨 크기 규칙
- 같은 반경 체계

차이는 채움, 외곽선, 맥락별 크기에서만 둔다.

AI CTA에 한해서는 아래 예외를 허용할 수 있다.

- 프라이머리 블루 기반 그라디언트
- 약한 glow
- 짧은 shimmer 또는 reveal 애니메이션

### 7.4 Secondary CTA와 칩

- 흰색 배경
- 검정 텍스트 또는 프라이머리 텍스트
- 보이는 외곽선
- 선택 상태는 프라이머리 블루 채움 또는 프라이머리 외곽선 강조로 표현

### 7.5 카드

- 흰색 둥근 표면
- 강한 제목 두께
- 클릭 가능하거나 중요할 때 얇은 외곽선 사용
- 반투명 유리 표현 금지

AI 카드나 AI 결과 요약 카드에는 아래 예외를 둘 수 있다.

- 연한 블루 톤 인셋
- 프라이머리 외곽선
- 미세한 glow
- 얕은 gradient overlay

### 7.6 리스트와 테이블

- 행 간 구분은 여백, 외곽선, 배경 톤 차이로 만든다
- 상태는 텍스트와 배지를 함께 사용한다
- 모바일에서는 카드형 목록 전환을 우선 고려한다
- 데스크톱에서는 밀도 높은 테이블로 확장할 수 있다
- 주요 행 액션은 프라이머리 블루로 빠르게 찾을 수 있어야 한다
- 리스트 item은 내용 높이만큼만 차지해야 하며, 부모 높이에 맞춰 불필요하게 늘어나면 안 된다
- 리스트가 짧을 때는 위에서부터 정렬하고, 남는 공간을 억지로 채우지 않는다
- 리스트가 길 때는 리스트 영역에만 스크롤을 붙이고, 카드 outer box의 row 정렬을 유지한다
- 빈 상태는 회색 muted 텍스트로 충분하며, 의미 없는 큰 일러스트나 과한 패널을 추가하지 않는다

### 7.7 진행 상태와 피드백

- 트랙은 밝은 회색
- 채움은 프라이머리 블루 또는 진한 회색
- 단계 라벨과 상태 문구는 색 없이도 읽혀야 한다
- 성공, 실패, 경고, 진행 중은 문구와 아이콘, 구조 차이로 구분한다

AI 처리 중 상태에는 아래 표현을 허용한다.

- 움직이는 gradient bar
- 은은한 glow pulse
- 분석 중, 생성 중, 변환 중을 보여 주는 단계형 animation

### 7.8 AI 경험 영역

다음과 같은 구역은 AI 경험을 직접 전달해야 하는 영역으로 간주할 수 있다.

- AI 생성 시작 영역
- AI 업로드 분석/문서 생성 로딩 화면
- AI 추천 또는 자동 완성 결과 요약 카드
- AI 비서 또는 AI 도움 패널

이 구역에서는 아래 표현을 허용한다.

- 프라이머리 블루 중심의 gradient
- 카드 가장자리 또는 배경 뒤쪽의 glow
- 짧고 목적이 분명한 animation
- 결과 도착 시 reveal transition

이 구역에서도 지켜야 할 제한:

- 본문 텍스트 뒤에서 강하게 움직이지 않는다
- 인터랙션 우선순위를 흐리지 않는다
- 동시에 너무 많은 효과를 겹치지 않는다
- 효과는 AI 기능이 끝난 뒤 더 차분한 상태로 수렴해야 한다

### 7.9 데이터 시각화와 차트

대시보드 차트는 장식이 아니라 실제 현장 판단을 돕는 도구다.
데이터와 그래픽의 대응이 불분명하면 신뢰가 떨어지므로, 반드시 실제 값 기반으로 그린다.

차트 규칙:

- 선, 영역, 격자, 축 라벨은 같은 좌표계에서 일관되게 계산한다
- 계획값과 실제값은 같은 scale과 같은 padding을 사용한다
- gap, hatch, delta 표현은 실제 두 값의 차이를 나타낼 때만 사용한다
- SVG에 `preserveAspectRatio="none"`을 쓸 경우 텍스트도 같이 늘어나 보일 수 있으므로 숫자와 축 라벨은 별도 HTML 레이어 또는 왜곡되지 않는 SVG 레이어로 분리한다
- 축 숫자는 작고 조용해야 한다. 데스크톱 차트의 축 라벨은 본문보다 작고 `ink-faint` 톤을 기본으로 한다
- 숫자 라벨은 가로로 늘어나 보이면 안 되며, monospace처럼 보이게 하려고 과도한 letter-spacing을 주지 않는다
- 차트 summary 숫자는 핵심 비교값만 보여주고, 축 라벨보다 커도 되지만 카드 제목보다 강하면 안 된다
- 데이터가 임시 seed여도 시각화는 실제 chart처럼 보이게 계산되어야 한다

공정률 차트 권장값:

- chart grid line: 얇은 회색
- planned line: 낮은 대비 회색
- actual line: primary blue
- y-axis label: `0.65rem` 이하 또는 SVG 기준 작은 px 값
- x-axis label: 균등 배치하되 텍스트 자체가 늘어나 보이지 않게 한다

### 7.10 공정표와 타임라인

공정표는 일반 카드 UI보다 더 높은 정보 밀도와 직접 조작성을 가진다.
차트형 작업 공간에서는 가독성과 조작 hit-area를 동시에 지켜야 한다.

공정표 규칙:

- 좌측 row panel과 우측 timeline은 하나의 frame 안에서 같은 높이 체계를 공유한다
- timeline header는 연/월/일 계층을 작고 명확하게 나눈다
- 오늘, 주말, 마일스톤은 배경 톤과 텍스트를 함께 바꿔 색상만으로 의존하지 않는다
- bar label은 긴 공정명을 고려해 ellipsis 또는 overflow 정책을 명확히 둔다
- summary bar, item bar, milestone은 모양과 색으로 역할이 구분되어야 한다
- drag, resize, connection 같은 편집 상태는 cursor, outline, z-index, shadow로 명확히 보여준다
- hover guide나 crosshair는 작업을 돕는 수준이어야 하며 본문 정보를 가리면 안 된다

## 8. Do's and Don'ts

### Do

- 회색 기반의 뉴트럴 캔버스 위에 프라이머리 블루 `#1e1888`를 일관되게 사용한다
- 계층은 두께, 여백, 외곽선, 면 분리로 만든다
- 읽기 영역은 단색 면 위에 둔다
- 첫 번째 핵심 액션이 즉시 보이게 만든다
- 모바일에서 가장 먼저 써도 불편하지 않게 설계한다
- AI 영역에서는 glow, gradation, animation을 목적 있게 사용한다
- 배경은 단색 회색으로 안정감 있게 유지한다
- 같은 row 안에서 비교되는 outer box는 높이와 패딩 리듬을 맞춘다
- 차트와 공정표는 실제 데이터 좌표와 시각 좌표가 일치하게 만든다
- 짧은 리스트는 위에서부터 정렬하고, 긴 리스트는 내부 스크롤로 처리한다

### Don't

- 일반 업무 화면 전체를 그라디언트와 glow로 덮지 않는다
- 색상 하나만으로 상태를 전달하지 않는다
- 배경에 점, 격자, 질감 패턴을 넣지 않는다
- 다른 성격의 버튼에 제각각 다른 기하 형태를 주지 않는다
- 모바일과 데스크톱을 전혀 다른 흐름으로 만들지 않는다
- AI 효과를 장식처럼 남용해서 가독성과 조작성을 해치지 않는다
- 숫자, 축 라벨, 날짜 라벨이 SVG 비율 변화 때문에 가로로 늘어나게 두지 않는다
- 리스트 item을 부모 높이에 맞춰 강제로 늘리지 않는다
- 상호작용하지 않는 count/meta 텍스트를 습관적으로 pill box로 감싸지 않는다
