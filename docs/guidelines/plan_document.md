# Plan 문서 작성 규정

`docs/plan` 문서는 길게 설명하는 문서보다, 바로 작업으로 옮길 수 있는 실행 문서여야 한다.  
핵심은 `무엇을 만들지`, `어떤 stage로 나눌지`, `각 stage를 닫으려면 무엇이 끝나야 하는지`가 빠르게 보이는 것이다.

---

## 문서 최소화 원칙

plan 문서는 구현을 열기 위한 최소 문서만 남겨야 한다. 문서를 많이 만드는 것이 목적이 아니라, 필요한 계약만 짧게 고정하고 나머지는 코드와 스크립트 산출물로 진행하는 것이 원칙이다.

- 새 문서는 꼭 필요한 경우에만 만든다
- 가능하면 계획 문서 자체를 업데이트한다
- 새로 만드는 문서는 feature 또는 plan 단위에서 최대 1개로 제한한다
- 세부 규칙, raw input 계약, 생성 규칙, validation 규칙은 여러 문서로 나누지 말고 한 문서에 함께 모은다
- 나머지는 문서보다 코드와 스크립트 산출물로 진행한다

---

## 1. 문서 시작 규칙

문서 첫 부분에는 아래 3가지를 꼭 넣는다.

- 기능 또는 흐름 이름
- 왜 이 문서가 필요한지 1~2문장
- 백틱으로 감싼 핵심 목표 문장 1개

---

## 2. Stage 기본 형식

모든 stage는 아래 순서를 기본으로 쓴다.

```md
## Stage N. 제목 [선택적 상태]

목표:
한두 문장 설명

커밋 메시지:
`type: summary`

결과물:
- 파일 또는 산출물

체크포인트:
- [ ] 세부 항목

완료 기준:
확인 가능한 상태 설명
```

지켜야 할 규칙:

- `목표`는 사용자 변화 기준으로 쓴다
- `결과물`은 새 문서보다 기존 문서 업데이트나 코드/스크립트 산출물을 우선 적는다
- `체크포인트`는 바로 구현 가능한 작업 단위로 쪼갠다
- `완료 기준`은 눈으로 확인 가능한 상태로 쓴다
- 마지막 stage에는 QA 또는 수용 기준 검증을 넣는다

---

## 3. 체크리스트 규칙

- 한 항목에는 한 가지 행동만 쓴다
- 설계, 데이터 계약, UI, 상태 로직은 가능하면 분리해서 쓴다
- `UI 개선하기` 같은 큰 문장 대신 실제 수정 내용을 쓴다
- 구현자가 보고 바로 파일 후보를 떠올릴 수 있어야 한다

좋은 예:

- [ ] CTA 클릭 시 overview session route가 열리게 만든다
- [ ] 현재 step의 `relatedNodeId`와 session page 안 Range Map 하이라이트를 연결한다
- [ ] 마지막 step 완료 시 추천 node를 선택한 상태로 대시보드에 복귀하게 만든다

---

## 4. Stage 분할 기준

보통은 아래 흐름을 기본으로 잡는다.

1. 목표와 원칙 고정
2. 데이터 계약 또는 카피 규칙 정의
3. 화면 구조와 상태 흐름 설계
4. 구현
5. 스타일/레이아웃/전환 다듬기
6. QA와 수용 기준 검증

모든 문서가 꼭 이 순서를 그대로 따라야 하는 것은 아니지만, 구현 전에 설계와 계약을 먼저 닫는 흐름은 유지한다.

---

## 5. Stage Sample

아래는 실제 plan 문서에 들어갈 수 있는 stage sample이다.

```md
## Stage 4. 구조 파악 세션 페이지 구조 설계

목표:
별도 overview session page에서 시퀀스가 어떻게 시작되고, 각 step이 어떻게 바뀌며, 끝났을 때 대시보드와 어디로 연결되는지 화면 구조를 고정한다.

커밋 메시지:
`docs: define overview session page architecture`

결과물:
- `docs/dev/overview_session_architecture.md`

체크포인트:
- [ ] `구조 파악하기 시작` 클릭 시 별도 overview session route로 이동한다고 정의한다
- [ ] session page 안에 `상단 안내 카드 + Range Map + 이전/다음/완료 조작` 구조를 둔다고 명시한다
- [ ] 현재 step의 `relatedNodeId`만 Range Map에서 하이라이트한다고 명시한다
- [ ] 하이라이트되지 않은 노드는 살짝 dimmed 처리해 시선을 분산시키지 않게 한다
- [ ] 마지막 step에서는 `이제 어디부터 들어가면 되는지`를 말해 주고, 완료 시 추천 node를 선택한 상태로 대시보드에 복귀하게 한다

완료 기준:
디자이너나 구현자가 별도 해석 없이도 overview session page의 시작, 진행, 완료, 대시보드 복귀 상태를 순서대로 그릴 수 있다.
```

## 6. 마지막 점검

새 plan 문서를 만들기 전에 아래만 확인하면 된다.

- [ ] 문서 첫 부분에 기능명, 배경, 핵심 목표 문장이 있다
- [ ] stage마다 `목표`, `커밋 메시지`, `체크포인트`, `완료 기준`이 있다
- [ ] 체크리스트가 실제 작업 단위로 쪼개져 있다
- [ ] 마지막에 QA 또는 수용 기준 검증 stage가 있다
- [ ] 기존 [dashboard_json_contract.md](/Users/leo/Desktop/fun/programming/bite_course/docs/contracts/dashboard_json_contract.md) 업데이트로 해결 가능한지 먼저 확인했다
- [ ] 새 dev 문서가 정말 필요하다면 1개만 만들고, 세부 규칙과 계약은 그 한 문서에 모았다
- [ ] 불필요한 문서, 파일 등을 생성하지 않고 핵심만 남아있다
