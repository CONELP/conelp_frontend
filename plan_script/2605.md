기준공정표버튼쪽에 과거 깃 히스토리 찾아봐. 기준공정표 히스토리를 보여주는 메뉴 코드가 있었거든,
지금 백엔드에서 is_main인 공정표를 여러개 보내줄거야.
그중에 가장 최신인 is_main버전이 기준공정표이고, 나머지는 과거 기준공정표야.
기준 공정표 버튼을 누르면 과거 기준공정표를 선택할 수 있게 해주고,
누르면 현재 기준공정표의 뒷배경으로 선택한 과거 기준공정표의 노드들이 보여지게 해줘.

변경된 api는 다음과같아.
# API 변경 이력 — 2026년 05월 13일 13:49

## 개요
공정표 버전 (`scheduleVersion`) 의 동작 방식 변경. `isMain` 은 더 이상 프로젝트당 1개로 강제되지 않으며 여러 행이 true 일 수 있고, 메인 지정 시각(`setMainAt`) 누적으로 메인 이력을 추적한다. 삭제는 soft delete (`isActive=false`) 로 변경되어 자식 데이터는 보존된다. 모든 CRUD 는 활성 버전만 대상. 프론트는 `setMainAt` 시간 순서로 "현재 메인" 을 구별한다.

## 수정 API

### `GET /api/scheduleVersion/getScheduleVersionList` — 응답 필드 확장, 활성 필터링
- 변경 전 응답 (200): `[{id, versionName, isMain}]`
- 변경 후 응답 (200): `[{id, versionName, isMain, setMainAt, createdAt, updatedAt}]`
- 응답 항목은 `isActive=true` 인 버전만 포함 (soft-deleted 버전은 응답에서 제외).
- `isMain=true` 인 행이 여러 개 존재할 수 있음. 프론트는 그중 `setMainAt` 이 가장 큰 1개를 "현재 메인" 으로 표시, 나머지 `isMain=true` 는 "과거 메인 이력", `isMain=false` 는 "working draft".
- 메인이 된 적 없는 버전의 `setMainAt` 은 `null`.

응답 예시:
```json
[
  {"id": 1, "versionName": "초안",   "isMain": true,  "setMainAt": "2026-04-01T10:00:00+09:00", "createdAt": "2026-03-20T...", "updatedAt": "2026-04-01T..."},
  {"id": 2, "versionName": "수정안", "isMain": true,  "setMainAt": "2026-05-10T14:30:00+09:00", "createdAt": "2026-05-01T...", "updatedAt": "2026-05-10T..."},
  {"id": 3, "versionName": "검토중", "isMain": false, "setMainAt": null,                        "createdAt": "2026-05-11T...", "updatedAt": "2026-05-11T..."}
]
```
→ id=2 가 현재 메인 (setMainAt 최신), id=1 은 과거 메인, id=3 은 working draft.

### `POST /api/scheduleVersion/createScheduleVersion` — 응답 필드 확장, 카운트 기준 변경
- 변경 전 응답 (200): `{id, versionName, isMain}`
- 변경 후 응답 (200): `{id, versionName, isMain: false, setMainAt: null, createdAt, updatedAt}`
- 변경 전 errors: 없음
- 변경 후 errors:
  - 400: "프로젝트당 최대 5개의 공정표 버전(메인 제외)만 생성할 수 있습니다."
- "최대 5개" 카운트 기준이 **전체 → working draft (`isMain=false && isActive=true`)** 로 변경. `isMain=true` 버전은 카운트에 포함되지 않음.

### `POST /api/scheduleVersion/duplicateScheduleVersion/{scheduleVersionId}` — 응답 필드 확장, 비활성 소스 차단
- 변경 전 응답 (200): `{id, versionName, isMain: false}`
- 변경 후 응답 (200): `{id, versionName, isMain: false, setMainAt: null, createdAt, updatedAt}`
- 변경 후 errors:
  - 400: "프로젝트당 최대 5개의 공정표 버전(메인 제외)만 생성할 수 있습니다."
  - 400: "ScheduleVersion not found"
  - 400: "비활성 공정표 버전은 복제할 수 없습니다."
- 카운트 기준은 createScheduleVersion 과 동일.

### `PUT /api/scheduleVersion/updateScheduleVersion/{scheduleVersionId}` — 응답 필드 확장, 차단 조건 추가
- 변경 전 응답 (200): `{id, versionName, isMain}`
- 변경 후 응답 (200): `{id, versionName, isMain, setMainAt, createdAt, updatedAt}`
- 변경 전 errors: 없음
- 변경 후 errors:
  - 400: "ScheduleVersion not found"
  - 400: "메인 공정표는 수정/삭제할 수 없습니다."  *(과거 메인 이력 포함 — `isMain=true` 인 모든 버전 차단)*
  - 400: "비활성 공정표 버전은 수정/삭제할 수 없습니다."
- 수정 성공 시 `updatedAt` 갱신.

### `DELETE /api/scheduleVersion/deleteScheduleVersion/{scheduleVersionId}` — soft delete 로 변경, 차단 조건 추가
- 변경 전: 물리 삭제 (CASCADE 로 자식 Work/WorkDependency/Milestone 등 동반 삭제).
- 변경 후: **soft delete** — `isActive=false` 로 마킹만. 자식 데이터는 물리적으로 보존.
- 변경 후 errors:
  - 400: "ScheduleVersion not found"
  - 400: "메인 공정표는 수정/삭제할 수 없습니다."  *(과거 메인 이력 포함)*
  - 400: "비활성 공정표 버전은 수정/삭제할 수 없습니다."  *(이미 soft-deleted 상태)*
  - 400: "최소 1개의 공정표 버전(메인 제외)은 유지해야 합니다."
- "최소 1개" 카운트 기준이 **전체 → working draft (`isMain=false && isActive=true`)** 로 변경.
- soft-deleted 된 버전은 모든 다운스트림 조회 (embedding 매칭, object3d, wTypConnector 등) 의 effective main 후보에서 자동 제외.

### `POST /api/scheduleVersion/setScheduleMain/{scheduleVersionId}` — 다중 main 누적, 비활성 차단
- 변경 전 응답 (200): `{id, versionName, isMain: true}`
- 변경 후 응답 (200): `{id, versionName, isMain: true, setMainAt, createdAt, updatedAt}`
- 변경 전 동작: 기존 `isMain=true` 행을 `false` 로 전환 후 target 을 `isMain=true` 로 (프로젝트당 메인 1개 강제).
- 변경 후 동작: **기존 `isMain=true` 행은 그대로 둠**. target 의 `isMain=true` + `setMainAt=NOW()` 갱신. 결과적으로 `isMain=true` 행이 여러 개가 될 수 있다.
- 프론트의 "현재 메인" 식별 = `getScheduleVersionList` 응답의 `isMain=true` 중 `setMainAt` 이 가장 큰 1개.
- 변경 전 errors (409): "메인 공정표가 다른 요청에 의해 이미 변경되었습니다." → **삭제** (`isMain` 다중 true 허용으로 unique 충돌 race 자체가 사라짐).
- 변경 후 errors:
  - 400: "Project ID가 필요합니다."
  - 400: "ScheduleVersion not found"
  - 400: "ScheduleVersion 이 호출 프로젝트 소속이 아닙니다."
  - 400: "비활성 공정표 버전은 메인으로 지정할 수 없습니다."
- 동일 `scheduleVersionId` 로 재호출 시 `setMainAt` 만 갱신, wtyp_connector swap 은 skip.

## 호환성 메모

- **`isMain` 의 의미 변화**: 프론트가 "이 버전이 현재 메인인가?" 를 `isMain === true` 로만 판정하던 코드는 더 이상 정확하지 않다. 응답에서 `isMain=true` 인 항목이 여러 개일 때, **`setMainAt` 이 가장 큰 1개만 "현재 메인"** 으로 표시하고 나머지는 "과거 메인 이력" 으로 분리 렌더링 권장.
- **`setMainAt = null`**: 한 번도 메인이 된 적 없는 버전. 정렬·표시 시 null 처리 주의.
- **삭제 결과**: 더 이상 물리 삭제가 아니므로 다른 화면 (Work 목록, Milestone 등) 이 즉시 비어 보이는 동작은 동일하지만, DB 단에서는 데이터가 남아 있다 (감사·복구 가능성을 위한 보존). 프론트 캐시 무효화 정책은 기존과 동일하게 유지.
- **에러 메시지 변경**: 4xx 응답의 body 메시지 텍스트가 일부 변경됨. 메시지 문자열로 분기하던 프론트 처리는 새 문구로 갱신 필요.
- **응답 신규 필드 (`setMainAt`, `createdAt`, `updatedAt`)**: 무시해도 동작에는 영향 없으나, 메인 이력 UI 를 새로 만들 경우 활용.
- **카운트 한도 변경**: 최대 5개 한도가 "전체" → "working draft" 로 바뀌었으므로, `isMain=true` 가 누적된 프로젝트는 working draft 5개를 추가로 보유 가능. 사용자 가이드 문구 갱신 필요.
